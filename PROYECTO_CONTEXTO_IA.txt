────────────────────────────────────────────────────────────────────────────────
1. RESUMEN EJECUTIVO
────────────────────────────────────────────────────────────────────────────────

"La Ruta del Sabor" (LRDScore) es un sistema de gestión integral para restaurantes,
desarrollado como una plataforma SaaS multi-tenant. Permite que múltiples
restaurantes (empresas) operen de forma aislada dentro de una sola instancia del
servidor, compartiendo la infraestructura pero manteniendo sus datos completamente
separados.

Objetivo principal: digitalizar y optimizar las operaciones de un restaurante
mediano, cubriendo el ciclo completo desde que el cliente hace un pedido hasta
que se procesa el pago (incluyendo pagos mixtos y cuentas divididas), la gestión
de cocina, inventario con costeo real, caja, comprobantes de venta y reportes.

Mercado objetivo: Restaurantes medianos en Perú, especialmente aquellos que aún
trabajan con papel o sistemas desconectados entre sí.

Propuesta de valor:
- Sistema unificado: mozo, cocina, caja y gerencia en una sola plataforma
- Tiempo real: la cocina y el mozo ven cambios de estado al instante vía SSE (sin polling)
- Multi-tenant con planes: un solo servidor atiende N restaurantes, cada uno con
  su propio plan de suscripción que habilita/restringe módulos (Módulo 0)
- Costos bajos: stack open source + PostgreSQL

Estado actual: backend con los Módulos 0 a 7 de la especificación implementados
(entitlements por plan, ciclo de pedidos con comandas y split de cuentas, doble
disponibilidad "86", costeo/margen real, comprobantes de venta con IGV). El
frontend, en cambio, solo cubre el flujo original de 5 pantallas — la UI para
split de pagos, agregar ítems, doble 86 y comprobantes de venta AÚN NO EXISTE
(ver sección 7 y 9). Entorno de trabajo: desarrollo local, zona horaria forzada
a America/Lima (GMT-5) tanto en el JVM como en la serialización JSON.

3.2 MULTI-TENANCY
──────────────────
  Todas las entidades operacionales extienden BaseTenantEntity, que contiene:
    - id (Long, PK auto-increment)
    - empresaId (Long, @TenantId — Hibernate inyecta filtro WHERE automático)
    - createdAt / updatedAt (LocalDateTime, auto-gestionados vía @PrePersist/@PreUpdate)

  Excepciones que NO extienden BaseTenantEntity (viven a nivel de plataforma, no de tenant):
  Empresa, Suscripcion, Plan, PlanModulo.

  El JWT incluye el claim "empresaId". Al autenticarse, JwtRequestFilter
  extrae ese claim y lo guarda en TenantContext (ThreadLocal). Hibernate lo usa
  automáticamente para filtrar todas las queries por empresa. El filtro limpia
  TenantContext en un bloque finally para evitar fugas entre requests del pool de Tomcat.

3.3 AUTENTICACIÓN, AUTORIZACIÓN Y ENTITLEMENTS (Módulo 0)
────────────────────────────────────────────────────────────
  Flujo de login:
    POST /api/auth/login  →  { correo, password }
    Response (AuthResponseDTO): { token, correo, rol, empresaId,
                                  modulosHabilitados: string[], estadoSuscripcion }

  El token JWT contiene: sub (correo), rol, empresaId, usuarioId, iat, exp.
  El rol ROLE_COCINA recibe un token de larga duración (30 días por defecto) para no
  interrumpir el servicio de la tablet/kiosco de cocina; el resto usa la expiración estándar (24h).

  Roles (RolUsuario):
    ROLE_SUPER_ADMIN — acceso completo, gestión de empresas y usuarios
    ROLE_GERENTE     — gestión operativa, reportes, sin crear empresas
    ROLE_CAJERO      — caja, cobro de pedidos, historial
    ROLE_MOZO        — crear/confirmar pedidos, entregar
    ROLE_COCINA      — solo KDS (ver y actualizar estados de preparación)

  Entitlements por plan (Módulo 0 — implementado):
    - Empresa → Suscripcion (ACTIVA | SUSPENDIDA | VENCIDA) → Plan → lista de PlanModulo.
    - Enum Modulo: PEDIDOS, CAJA, INVENTARIO, KDS (core=true) y REPORTES_AVANZADOS,
      FACTURACION, RESERVAS, FIDELIZACION (core=false, premium).
    - @RequiereModulo(Modulo.X) en un endpoint dispara ModuloInterceptor, que revisa
      la suscripción vigente del tenant (vía TenantContext) independientemente de @PreAuthorize.
    - Suscripción VENCIDA: los módulos core quedan en solo lectura (GET/HEAD permitidos,
      escritura lanza SuscripcionVencidaException); los módulos premium quedan bloqueados
      (ModuloNoHabilitadoException) sin importar el método HTTP.
    - Sin suscripción o SUSPENDIDA: cualquier endpoint anotado lanza ModuloNoHabilitadoException.
    - El login ya devuelve modulosHabilitados y estadoSuscripcion para que el frontend
      oculte/deshabilite UI — pero el frontend actual (LRDS_Front) todavía no consume estos
      dos campos en ninguna pantalla.

3.4 TIEMPO REAL (SSE)
──────────────────────
  SseEmitterManager (@Component, singleton):
    - Indexado en dos niveles: ConcurrentHashMap<empresaId, ConcurrentHashMap<usuarioId, sesión>>.
      Cada sesión guarda el rol del usuario y una CopyOnWriteArrayList<SseEmitter> (multi-tab).
    - publicarTenant(empresaId, evento, datos)          → broadcast a todo el tenant
    - publicarUsuario(empresaId, usuarioId, evento, datos) → a un usuario específico
    - publicarPorRol(empresaId, rol, evento, datos)      → a todos los de un rol en el tenant
    - Limpieza atómica de emitters/sesiones muertas en onCompletion/onTimeout/onError.

  Eventos publicados:
    "NUEVO_PEDIDO"              — mozo confirma pedido (BORRADOR → RECIBIDO)
    "EN_PREPARACION"            — cocina inicia preparación
    "PEDIDO_LISTO"              — cocina marca pedido listo (al mozo + rol COCINA)
    "NUEVA_COMANDA"             — Módulo 4: se agregan ítems a un pedido ya confirmado
    "PEDIDO_REQUIERE_REVISION"  — al rol GERENTE: stock insuficiente detectado en preparación
    "PRODUCTO_AGOTADO"          — Módulo 6: producto marcado AGOTADO_TEMPORAL/SERVICIO
    "PRODUCTO_DISPONIBLE"       — Módulo 6: producto revertido a DISPONIBLE
    "RESET_DISPONIBILIDAD"      — Módulo 6: reseteo masivo de AGOTADO_SERVICIO al cerrar caja
    "AVISO_PEDIDO_LISTO"        — escalación server-side (EscalacionScheduler, cada 30s):
                                   t≥1min → todos los MOZO del tenant
                                   t≥2min → todos los CAJERO del tenant
                                   t≥5min → GERENTE y SUPER_ADMIN del tenant

  Endpoint SSE: GET /api/kds/eventos (acepta ?token= para EventSource, que no soporta headers).
  El empresaId/usuarioId/rol del emitter provienen SIEMPRE del JWT validado, nunca de un
  parámetro del request (invariante de seguridad documentado en el propio código).

  Nota: /api/pedidos/{id}/notificacion/ack existe pero es un endpoint informacional (ACK)
  que no detiene la cascada de escalación — la escalación corre por tiempo del lado del
  servidor independientemente de si el mozo confirmó haber visto el aviso.

5. FLUJO DE NEGOCIO PRINCIPAL
────────────────────────────────────────────────────────────────────────────────

5.1 CICLO DE VIDA DE UN PEDIDO (estado agregado + estado por ítem)
────────────────────────────────────────────────────────────────────
  El pedido tiene un estado_actual "agregado" que PedidoServiceImpl/KdsServiceImpl
  recalculan a partir de los EstadoItem de sus PedidoDetalle activos (no cancelados):
    - todos EN_PREPARACION (alguno) → EN_PREPARACION
    - si no, alguno PENDIENTE       → RECIBIDO
    - si no, alguno LISTO           → LISTO
    - si no (todos ENTREGADO)       → ENTREGADO
    - si todos CANCELADO            → CANCELADO

  [MOZO]
    1. Crea pedido (POST /api/pedidos) → BORRADOR. Ítems en PENDIENTE, numero_comanda=1.
       No afecta inventario. El backend fija el precio desde el catálogo (nunca confía
       en el precio que mande el cliente).
    2. Confirma (PUT /api/pedidos/{id}/confirmar) → RECIBIDO.
       - Rechaza si algún producto del pedido no está DISPONIBLE (Módulo 6).
       - Reserva insumos por receta (reservarInsumosParaPedido): SOLO incrementa
         stock_reservado, NO toca stock_actual. Si falta stock lanza
         StockInsuficienteException con el detalle exacto de insumos faltantes.
       - SSE "NUEVO_PEDIDO" a todo el tenant.

  [COCINA — KDS]
    3. Toma la orden (PUT /api/kds/{id}/preparando) → ítems PENDIENTE pasan a EN_PREPARACION.
       *** AQUÍ ocurre el consumo REAL de inventario *** (convertirItemsAConsumo):
       descuenta stock_actual, libera el stock_reservado correspondiente, registra
       KardexMovimiento CONSUMO_PRODUCCION, y congela el costo_unitario_consumido en
       cada PedidoDetalle (snapshot para reportes de margen estables — Módulo 5).
       Si el stock físico no alcanza pese a la reserva, marca pedido.requiereRevision=true
       y notifica al GERENTE por SSE, en vez de fallar en silencio.
       SSE "EN_PREPARACION" a todo el tenant.
    4. Marca listo (PUT /api/kds/{id}/listo) → ítems EN_PREPARACION pasan a LISTO.
       SSE "PEDIDO_LISTO" al mozo dueño del pedido + rol COCINA.
       A partir de aquí, EscalacionScheduler empieza a contar minutos para escalar
       el aviso si nadie entrega (ver sección 3.4).

  [MOZO]
    5. Entrega (PUT /api/pedidos/{id}/entregar) → exige que todos los ítems activos
       estén LISTO o ENTREGADO; los LISTO pasan a ENTREGADO.

  [CAJERO] — dos vías, NO deben mezclarse en la misma mesa (ver hallazgos, sección 12)
    6a. Pago directo (POST /api/pedidos/{id}/pagar): pedido debe estar LISTO/ENTREGADO;
        valida que la suma de pagos (array multi-método) cubra el total; pedido → PAGADO.
    6b. Split de cuentas — Módulo 4 (POST .../documentos-cobro y .../documentos-cobro/{id}/pagar):
        se crean uno o más DocumentoCobro (por ÍTEMS específicos o por MONTO libre);
        cada uno se paga independientemente; el pedido pasa a PAGADO recién cuando la
        suma de documentos PAGADO cubre el total del pedido.
    Nota: el descuento de inventario NO ocurre aquí — ya ocurrió en el paso 3
    (EN_PREPARACION). El pago solo mueve dinero y cierra el ciclo del pedido.

  [GERENTE]
    7. Dashboard con KPIs (GET /api/reportes/dashboard), margen de utilidad real
       (GET /api/reportes/margen — requiere módulo REPORTES_AVANZADOS) y alertas de
       stock bajo (GET /api/inventario/alertas).

5.2 MÓDULO 4 — CICLO DE VIDA AVANZADO (comandas múltiples, cancelaciones, split)
──────────────────────────────────────────────────────────────────────────────────
  - Agregar ítems a un pedido ya confirmado (POST /api/pedidos/{id}/items): crea una
    nueva "comanda" (numero_comanda incremental), reserva SOLO el inventario de los
    ítems nuevos, y si el pedido estaba LISTO/ENTREGADO lo retrocede a RECIBIDO
    (porque ahora tiene ítems pendientes otra vez). Dispara SSE "NUEVA_COMANDA".
  - Cancelar un ítem específico (PUT /api/pedidos/{id}/items/{detalleId}/cancelar):
      · Si estaba PENDIENTE           → libera la reserva de inventario.
      · Si estaba EN_PREPARACION/LISTO → el consumo ya se registró; se asume merma
        implícita (no se devuelve stock).
      · Si estaba ENTREGADO           → exige rol GERENTE + motivo obligatorio (auditoría).
  - Split de cuentas: ver paso 6b arriba. Un DocumentoCobro tipo "ITEMS" selecciona
    PedidoDetalle específicos; tipo "MONTO" es un monto libre (ej. dividir la cuenta
    en partes iguales sin atarlo a ítems concretos).

5.3 MÓDULO 6 — DISPONIBILIDAD "DOBLE 86"
────────────────────────────────────────────
  - AGOTADO_TEMPORAL: cocina lo marca (PUT /api/kds/productos/{id}/agotado-temporal),
    reversible en cualquier momento por cocina (PUT .../disponible).
  - AGOTADO_SERVICIO: cocina lo marca (PUT .../agotado-servicio) cuando el insumo no
    volverá hasta el día siguiente; NO se puede revertir manualmente — solo se
    restablece automáticamente al cerrar caja (ver 5.4).
  - Confirmar un pedido con productos no DISPONIBLE está bloqueado (ver 5.1 paso 2).
  - GET /api/kds/productos/porciones calcula, por receta, cuántas porciones más se
    pueden servir de cada producto (mínimo de stock_disponible / cantidad_requerida
    entre todos sus insumos) — útil para advertir al mozo antes de que se agote.

5.4 CICLO DE CAJA
──────────────────
  - Abrir (POST /api/caja/abrir): un cajero no puede tener dos sesiones ABIERTA a la vez.
  - Durante el turno, monto_final_calculado = monto_inicial + Σ pagos EFECTIVO
    (YAPE/TARJETA/PLIN/TRANSFERENCIA NO entran al conteo físico).
  - Cerrar (PUT /api/caja/cerrar/{id}): recibe monto_final_declarado (conteo físico);
    diferencia se calcula en la BD (columna GENERATED). Al cerrar, TODOS los productos
    en AGOTADO_SERVICIO del tenant se resetean a DISPONIBLE automáticamente y se emite
    SSE "RESET_DISPONIBILIDAD".

5.5 MÓDULO 5 — COSTEO Y MARGEN (ya implementado, contradice versiones previas de este doc)
────────────────────────────────────────────────────────────────────────────────────────────
  GET /api/reportes/margen (requiere módulo REPORTES_AVANZADOS) calcula, para un rango
  de fechas: ingresos efectivos (prorrateando el descuento del pedido), costo de ventas
  (a partir de costo_unitario_consumido, NUNCA recalculado retroactivamente), utilidad
  bruta y margen %, desglosado por producto y por categoría, más el costo de la merma
  (ítems cancelados que ya habían consumido inventario). Los productos sin receta usan
  costo_referencial como estimado (esEstimado=true en el desglose).

5.6 MÓDULO 7 — COMPROBANTES DE VENTA (parcial, sin integración SUNAT real)
───────────────────────────────────────────────────────────────────────────
  POST /api/documentos-venta emite NOTA_VENTA (sin gate, sin IGV desglosado),
  BOLETA o FACTURA (ambas requieren módulo FACTURACION; FACTURA exige RUC de 11
  dígitos válido). El IGV (18%) se calcula por división del total (que ya lo incluye)
  usando el divisor 1.18. El correlativo es atómico vía INSERT ... ON CONFLICT DO
  UPDATE sobre una tabla series_correlativo (por empresa_id + tipo), sin bloqueos de
  aplicación. La anulación (PUT .../anular) es un cambio de estado + motivo — el
  comprobante nunca se borra. No hay generación de XML ni envío a un OSE/SUNAT real;
  los estados ENVIADO_SUNAT/ACEPTADO/RECHAZADO existen en el enum pero ningún flujo
  los setea todavía.

8. SEGURIDAD
────────────────────────────────────────────────────────────────────────────────

- Contraseñas hasheadas con BCrypt (mínimo 8 caracteres, validado en backend).
- JWT firmado HMAC-SHA256; claims sub/rol/empresaId/usuarioId/iat/exp.
- Token de 24h estándar, 30 días para ROLE_COCINA (cuenta kiosco).
- @JsonIgnoreProperties({"passwordHash"}) en Usuario — nunca serializa el hash.
- @PreAuthorize por endpoint + ModuloInterceptor por entitlements (dos capas
  independientes: pasar una no implica pasar la otra).
- TenantContext como ThreadLocal, limpiado en finally del JwtRequestFilter.
- Fallback ?token= para SSE, solo cuando no hay header Authorization.
- Cambio de contraseña self-service exige la contraseña actual; el reset admin no.
- CORS: orígenes localhost/127.0.0.1 en cualquier puerto + dominio de Vercel de
  producción, credentials habilitado.

Cosas pendientes SOLO relevantes para producción (no bloquean el desarrollo local):
  - JWT_SECRET tiene valor por defecto hardcodeado en application.properties.
  - No hay refresh token (expiración fija).
  - No hay rate limiting en /api/auth/login.
  - No hay HTTPS configurado (se asumiría un reverse proxy en producción).

Hallazgos de correctness (sí relevantes en cualquier entorno, incluido local):
  - No existe ningún @Lock/@Version en el proyecto: el descuento de inventario
    (convertirItemsAConsumo) y el cierre de split de cuentas (pagarDocumentoCobro,
    que suma documentos PAGADO y compara contra el total) son "leer, calcular,
    guardar" sin bloqueo — con dos requests concurrentes podría haber una
    condición de carrera. No se ha priorizado porque el volumen de uso actual
    (desarrollo/pruebas) no lo dispara, pero es una tarea pendiente real.
  - El intento de eliminar un ítem cancelado ya está scopeado correctamente al
    pedido (busca dentro de pedido.getDetalles(), no un findById suelto), así que
    el IDOR cross-tenant mencionado en auditorías previas de este proyecto ya NO
    reproduce en el código actual.

12. DECISIONES TÉCNICAS CLAVE
────────────────────────────────────────────────────────────────────────────────

1. Multi-tenancy por columna vs esquema:
   Columna (empresa_id) sobre esquema separado, por simplicidad y compatibilidad
   nativa con @TenantId de Hibernate. Menor aislamiento físico, suficiente para el
   tamaño del mercado objetivo.

2. Entitlements desacoplados de roles (Módulo 0):
   @PreAuthorize (rol) y ModuloInterceptor (plan/suscripción) son dos chequeos
   independientes y ambos deben pasar. Permite vender el mismo rol (ej. GERENTE)
   con distintas capacidades según el plan contratado, sin tocar la matriz de roles.

3. Timing del consumo de inventario — CAMBIÓ respecto al diseño original:
   El consumo real (descuento de stock_actual) ya NO ocurre al pagar; ocurre cuando
   cocina marca EN_PREPARACION. Esto permite: (a) que Módulo 6 (doble 86) tenga
   sentido — el producto se "gasta" cuando se cocina, no cuando se cobra — y
   (b) detectar quiebres de stock físico antes del cobro, marcando requiereRevision
   en vez de fallar silenciosamente. La reserva (Módulo 3, al confirmar) sigue
   existiendo para bloquear stock lógico entre el pedido y la cocina.

4. Snapshot de costo por ítem (costo_unitario_consumido):
   Se congela en EN_PREPARACION y nunca se recalcula. Si el costo del insumo sube
   mañana, los reportes de margen de pedidos ya cocinados no se alteran retroactivamente.

5. SSE sobre WebSocket, ahora con indexado por rol:
   Unidireccional y suficiente para notificar cambios de estado. El indexado por
   (empresa, usuario) + rol permite tanto notificar a un usuario puntual como a
   "todos los cajeros del tenant" (escalación) sin canales separados.

6. Correlativo de comprobantes vía INSERT ... ON CONFLICT DO UPDATE:
   Evita huecos en la numeración sin necesidad de bloqueos de aplicación, incluso
   con múltiples nodos del backend corriendo en paralelo (una sola sentencia SQL
   atómica sobre PostgreSQL resuelve la concurrencia).

7. Sin ORM con esquema automático (ddl-auto=none):
   La base de datos se gestiona con scripts SQL versionados — control total sobre
   índices, vistas, columnas generadas (diferencia en sesiones_caja) y constraints.

8. Pago multi-método y split de cuentas conviven pero no deben mezclarse:
   Un pedido puede pagarse por la vía directa (/pagar) o por documentos de cobro
   (split), pero mezclar ambos flujos en la misma mesa genera inconsistencias
   lógicas — es una decisión de UI, no del backend, mantenerlos separados.

9. Soft delete universal:
   estado_registro=false en lugar de DELETE físico, preserva integridad referencial
   e historial para reportes y auditoría.

10. Zona horaria forzada a America/Lima:
    TimeZone.setDefault() en el arranque de CoreApplication (antes de que Spring
    levante el contexto) + spring.jackson.time-zone=America/Lima, para que
    ZoneId.systemDefault() (usado en timestamps y en el filtro de reportes) y la
    serialización JSON sean consistentes con la zona horaria del mercado objetivo.
