1. RESUMEN EJECUTIVO
────────────────────────────────────────────────────────────────────────────────

"La Ruta del Sabor" (LRDScore) es un sistema de gestión integral para restaurantes,
desarrollado como una plataforma SaaS multi-tenant. Permite que múltiples
restaurantes (empresas) operen de forma aislada dentro de una sola instancia del
servidor, compartiendo la infraestructura pero manteniendo sus datos completamente
separados.

Objetivo principal: digitalizar y optimizar las operaciones de un restaurante
mediano, cubriendo el ciclo completo desde que el cliente hace un pedido hasta
que se procesa el pago, incluyendo la gestión de cocina, inventario, caja y reportes.

Mercado objetivo: Restaurantes medianos en Perú, especialmente aquellos que aún
trabajan con papel o sistemas desconectados entre sí.

Propuesta de valor:
- Sistema unificado: mozo, cocina, caja y gerencia en una sola plataforma
- Tiempo real: la cocina ve pedidos al instante vía SSE (sin polling)
- Multi-tenant: un solo servidor puede atender N restaurantes
- Costos bajos: stack open source + PostgreSQL

Estado actual: MVP funcional con backend completo y frontend operativo para
todos los roles del restaurante.

3.2 MULTI-TENANCY
──────────────────
  Todas las entidades del dominio extienden BaseTenantEntity, que contiene:
    - id (Long, PK auto-increment)
    - empresaId (Long, @TenantId — Hibernate inyecta filtro WHERE automático)
    - createdAt / updatedAt (LocalDateTime, auto-gestionados)

  El JWT incluye el claim "empresaId". Al autenticarse, JwtRequestFilter
  extrae ese claim y lo guarda en TenantContext (ThreadLocal). Hibernate lo usa
  automáticamente para filtrar todas las queries por empresa.

  Resultado: una query SELECT * FROM pedidos se convierte automáticamente en
  SELECT * FROM pedidos WHERE empresa_id = ? sin código adicional.

3.3 AUTENTICACIÓN Y AUTORIZACIÓN
──────────────────────────────────
  Flujo de login:
    POST /api/auth/login  →  { correo, password }
    Response: { token, correo, rol, empresaId }

  El token JWT contiene: sub (correo), empresaId, rol, iat, exp

  JwtRequestFilter:
    1. Lee header "Authorization: Bearer <token>"
    2. Fallback: lee query param "?token=<jwt>" (necesario para EventSource/SSE)
    3. Valida token, carga UserDetails, establece SecurityContext
    4. Extrae empresaId → TenantContext

  Roles y permisos:
    ROLE_SUPER_ADMIN — acceso completo, gestión de empresas y usuarios
    ROLE_GERENTE     — gestión operativa, reportes, sin crear empresas
    ROLE_CAJERO      — caja, cobro de pedidos, historial
    ROLE_MOZO        — crear/confirmar pedidos, entregar
    ROLE_COCINA      — solo KDS (ver y actualizar estados de preparación)

3.4 TIEMPO REAL (SSE)
──────────────────────
  SseEmitterManager (@Component, singleton):
    - Mantiene lista CopyOnWriteArrayList<SseEmitter> (thread-safe)
    - suscribir(): crea un emitter, registra limpieza en completion/timeout/error
    - publicar(evento, datos): broadcast a todos los emitters activos

  Eventos publicados:
    "NUEVO_PEDIDO"    — cuando mozo confirma pedido (BORRADOR → RECIBIDO)
                        payload: { pedidoId, mesa, estado: "RECIBIDO" }
    "EN_PREPARACION"  — cuando cocina inicia preparación
                        payload: { pedidoId, mesa, estado: "EN_PREPARACION" }
    "PEDIDO_LISTO"    — cuando cocina marca pedido listo
                        payload: { pedidoId, mesa, estado: "LISTO" }

  Endpoint SSE: GET /api/kds/eventos  (acepta ?token= para EventSource)

  Frontend:
    KdsPage  escucha "NUEVO_PEDIDO" → recarga lista de pendientes
    MozoPage escucha "PEDIDO_LISTO" y "EN_PREPARACION" → actualiza tarjetas

5. FLUJO DE NEGOCIO PRINCIPAL
────────────────────────────────────────────────────────────────────────────────

5.1 CICLO DE VIDA DE UN PEDIDO
───────────────────────────────
  [MOZO]
    1. Crea pedido con productos y cantidades → estado: BORRADOR
       POST /api/pedidos
    2. Cuando está completo, lo envía a cocina → estado: RECIBIDO
       PUT /api/pedidos/{id}/confirmar
       → SSE "NUEVO_PEDIDO" broadcast a todos los clientes

  [COCINA — KDS]
    3. Ve el pedido en columna "NUEVOS", lo toma → estado: EN_PREPARACION
       PUT /api/kds/{pedidoId}/preparando
       → SSE "EN_PREPARACION" broadcast
    4. Termina de preparar → estado: LISTO
       PUT /api/kds/{pedidoId}/listo
       → SSE "PEDIDO_LISTO" broadcast

  [MOZO]
    5. Ve la tarjeta en verde "¡Listo!", lo lleva a la mesa → estado: ENTREGADO
       PUT /api/pedidos/{id}/entregar

  [CAJERO]
    6. Ve pedido en su pantalla (ENTREGADO), abre modal de cobro
    7. Selecciona métodos de pago (puede ser mixto, ej: S/10 Yape + S/5 Efectivo)
       POST /api/pedidos/{id}/pagar  →  { sesionCajaId, pagos: [...] }
    8. Sistema registra transacciones, descuenta stock de insumos (recetas),
       acumula monto_final_calculado en sesión de caja → estado: PAGADO

  [GERENTE]
    9. Visualiza dashboard con KPIs del período y alertas de stock
       GET /api/reportes/dashboard?inicio=&fin=
       GET /api/inventario/alertas

5.2 FLUJO DE INVENTARIO (automático al pagar)
──────────────────────────────────────────────
  Cuando un pedido se paga:
    - Por cada producto en los detalles del pedido
    - Si el producto tiene receta (receta_detalles)
    - Se descuenta la cantidad de cada insumo de la receta × cantidad pedida
    - Se registra un KardexMovimiento de tipo CONSUMO_PRODUCCION
    - Si el stock cae por debajo de stock_minimo → aparece en alertas

5.3 CICLO DE CAJA
──────────────────
  Cajero abre caja con monto inicial (fondo de cambio)
    POST /api/caja/abrir  →  { montoInicial }
    → SesionCaja en estado ABIERTA

  Durante el turno: los pagos se acumulan en monto_final_calculado

  Al cerrar el turno, cajero cuenta físicamente el dinero e ingresa el total
    PUT /api/caja/cerrar/{id}  →  { montoFinalDeclarado }
    → diferencia = declarado - calculado (cuadre de caja)
    → SesionCaja en estado CERRADA

8. SEGURIDAD
────────────────────────────────────────────────────────────────────────────────

- Contraseñas hasheadas con BCrypt (no se almacena texto plano)
- JWT firmado con HMAC-SHA256, claim empresaId incluido
- @JsonIgnoreProperties({"passwordHash"}) en entidad Usuario — nunca serializa el hash
- @PreAuthorize en cada endpoint con lista explícita de roles permitidos
- TenantContext como ThreadLocal — limpiado en finally block del filtro
- Fallback ?token= para SSE limitado solo cuando no hay header Authorization
- Separación de endpoints de contraseña: self-service vs admin-reset

Vulnerabilidades conocidas y pendientes de producción:
  - JWT_SECRET tiene valor por defecto inseguro (cambiar antes de producción)
  - No implementado refresh token (expiración fija 24h)
  - No implementado rate limiting en /api/auth/login
  - No implementado HTTPS (debe configurarse en nginx/reverse proxy)

10. MODELO DE NEGOCIO (SaaS)

Modelo actual (hipotético):
  El sistema está diseñado para ser multi-tenant desde su arquitectura base.
  Un solo despliegue puede atender múltiples restaurantes con datos aislados.

Posibles modelos de monetización a evaluar:
  - Suscripción mensual por restaurante (modelo SaaS clásico)
    Ej: S/99/mes básico, S/199/mes con módulos avanzados
  - Freemium: versión gratuita limitada (1 caja, 5 productos) + pago por upgrades
  - Por volumen de transacciones (% pequeño sobre cada pedido procesado)
  - Setup fee + mensualidad (común en mercado peruano de software empresarial)
  - Licencia perpetua para restaurantes que quieran servidor propio

Segmentación potencial de clientes:
  Tier 1: Restaurante pequeño (1 local, 1-2 mozos, 1 cajero)
  Tier 2: Restaurante mediano (1-3 locales, equipo completo)
  Tier 3: Cadena (N locales, cada uno como tenant separado, reportes consolidados)

Competidores referentes en Perú:
  - Rest (software local)
  - Alegra POS
  - El Mandado
  - Waiter (app mozo)
  - Sistemas custom hechos a medida

12. DECISIONES TÉCNICAS CLAVE
────────────────────────────────────────────────────────────────────────────────

1. Multi-tenancy por columna vs esquema:
   Se eligió columna (empresa_id) sobre esquema separado por simplicidad de
   migración y compatibilidad nativa con @TenantId de Hibernate 6.
   Trade-off: menor aislamiento físico, pero suficiente para el tamaño del mercado.

2. SSE sobre WebSocket:
   SSE es unidireccional (server → client) y suficiente para los casos de uso
   (notificar pedidos listos). Es más simple de implementar y escala bien con
   un load balancer sticky-session. WebSocket sería necesario solo si el cliente
   necesita enviar mensajes al servidor en tiempo real.

3. Sin ORM con esquema automático (ddl-auto=none):
   La base de datos se gestiona con scripts SQL versionados. Esto da control
   total sobre índices, vistas (vw_kds_cocina, vw_dashboard_ventas), columnas
   generadas (diferencia en sesiones_caja) y constraints.

4. Gráfico SVG sin librería:
   DashboardPage implementa un gráfico de barras con SVG puro para evitar
   agregar dependencias pesadas (Recharts, Chart.js). Trade-off: menos features,
   pero bundle más pequeño.

5. Pago multi-método:
   Una sola transacción de pedido puede tener múltiples TransaccionPago.
   Esto permite dividir el pago (Yape S/10 + Efectivo S/5.50) sin complejidad
   extra. El total se valida en el frontend antes de enviar.

6. Soft delete universal:
   Todos los recursos usan estado_registro=false en lugar de DELETE físico.
   Preserva integridad referencial e historial para reportes.
